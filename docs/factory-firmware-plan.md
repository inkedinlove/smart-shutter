# Factory Firmware Plan

Factory firmware is the missing piece that turns Smart Shutter into a QR-first,
Wi-Fi-first customer setup product.

## Factory And Recovery Firmware Image

The device should ship with a factory firmware image already installed.

That image should also serve as the recovery baseline for USB flashing when a
device needs bench repair or rework.

## Unique Device Identity

Each device should have a stable identity from first boot, such as:

- device ID
- claim code or claim URL generated by the platform
- optional printed short code for packaging

The device identity should be separate from the friendly label customers see in
the app.

## Setup Access Point

If the device has no saved Wi-Fi credentials, factory firmware should:

1. start a setup AP
2. expose a captive portal
3. let the customer enter Wi-Fi credentials locally

This keeps Wi-Fi setup on the device itself instead of pushing credentials
through the customer web app.

## WiFi Credential Storage

Wi-Fi credentials should be:

- collected only through the device's local setup portal
- stored locally on the device
- reused on reboot
- replaceable by re-entering setup mode

They should not be embedded in QR codes or exposed through the main app.

## Cloud Pairing

After Wi-Fi setup succeeds, factory firmware should:

- connect to the cloud
- publish online status
- make the claimed device visible to the customer in `/connect`
- identify itself with a stable per-device MQTT client identity

The claim flow and the online device status flow should stay separate but
connected:

- claim establishes ownership
- Wi-Fi setup brings the device online

## Per-Device Credential Direction

The current MVP can still use shared HiveMQ credentials, but factory firmware
should move toward device-specific cloud identity over time.

Near-term transition:

- keep shared broker credentials
- derive a per-device MQTT client ID
- track credential mode and status in the cloud

Long-term direction:

- provision per-device credentials at factory time
- support credential rotation and revocation
- move to AWS IoT Core X.509 certificates for production

## OTA After Online

Once online, the device should be able to:

- report firmware version
- request or receive update instructions
- download new firmware over Wi-Fi
- verify hash and later signature
- install and reboot safely

## Why This Removes Most USB Burden

With factory firmware in place:

- customers do not need USB for normal onboarding
- customers do not need Arduino tools
- Wi-Fi setup becomes self-service
- OTA becomes the normal update path

USB can then stay limited to:

- factory programming
- hardware recovery
- diagnostics on spare or returned devices
